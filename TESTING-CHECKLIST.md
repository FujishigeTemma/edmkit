# Testing Checklist

このチェックリストは、`edmkit` のテストレビューを始める前に観点を揃えるためのものです。
主に `tests/` 配下の pytest ベースのテストを対象とし、数値計算・時系列・EDM アルゴリズムの特性を前提にしています。

## 使い方

- まず `Must` を満たしているか確認する
- 次に `Should` を見て、保守性と説明可能性を揃える
- 最後に `Nice to have` を見て、改善候補を拾う
- 1件ごとのレビューでも、ファイル単位の棚卸しでも使ってよい

## Must

### 1. テストの目的

- [ ] そのテストが何を保証するかが明確である
- [ ] 1つのテストに複数の責務を詰め込みすぎていない
- [ ] unit / property / trend / integration / smoke のどれかが判別できる
- [ ] 下位レイヤで十分なものを不必要に高コストなレイヤで検証していない

### 2. 入力データの作り方

- [ ] 入力がテストしたい仕様を表している
- [ ] ランダム入力を使う場合、テスト内で seed を固定している
- [ ] 同じ入力生成コードが複数箇所にコピペされていない
- [ ] 共有 fixture を使う理由がある
- [ ] inline で十分な小さな例を不必要に fixture 化していない

### 3. カバー範囲

- [ ] 正常系だけでなく異常系もある
- [ ] 最小入力や境界値を押さえている
- [ ] 定数入力、短い系列、最小次元などの退化ケースが必要に応じて含まれている
- [ ] shape、dtype、値域、有限性などの基本契約が必要に応じて検証されている

### 4. アサーションの強さ

- [ ] アサーションが弱すぎない
- [ ] `shape` や `isfinite` だけで終わっていない
- [ ] 浮動小数点比較で exact match を乱用していない
- [ ] `atol` / `rtol` や閾値に理由がある
- [ ] 少し壊れた実装なら落ちるテストになっている

### 5. 数値計算・EDMとしての妥当性

- [ ] deterministic に確認できるものは exact test か example-based test で見ている
- [ ] 実装に依らない性質は property test で見ている
- [ ] 厳密値を持ちにくい現象は trend test で見ている
- [ ] trend test は単調傾向、順位、符号などを見る形になっている
- [ ] カオス系やノイズ系のテストが過度に brittle でない

### 6. 命名と構造

- [ ] テスト名が「何をしたとき、何が起きるか」を表している
- [ ] `test_case1` のような意味の薄い名前がない
- [ ] クラスやファイルのグルーピング基準が一貫している
- [ ] モジュール責務とテストファイルの切り方が対応している

### 7. 実行安定性

- [ ] テストが実行順序に依存していない
- [ ] 外部状態、グローバル状態、時刻、乱数に暗黙依存していない
- [ ] 重いテストには `slow`、GPU 専用には `gpu` が適切に付いている
- [ ] `smoke_test.py` は dev 依存や `tests` ヘルパーに依存していない

## Should

### 8. テストの粒度

- [ ] 1テスト1振る舞いの原則がおおむね守られている
- [ ] 似た確認は `parametrize` で整理されている
- [ ] ただし、読みやすさを損なう過度な抽象化は避けている

### 9. fixture と共有化

- [ ] `tests/conftest.py` に置く共有 fixture は複数ファイルで使われている
- [ ] fixture のスコープが妥当である
- [ ] 重い fixture が module scope 以上になっている理由がある
- [ ] fixture 名だけでデータの意味が分かる

### 10. property-based test の品質

- [ ] Hypothesis は本当に性質を検証する場面に使っている
- [ ] `assume(...)` に頼りすぎていない
- [ ] `NaN` / `inf` を除外するか含めるかが明示的である
- [ ] shape preservation、順序不変性、対称性、自己一致性、凸結合性などを狙えている

### 11. 異常系テストの読みやすさ

- [ ] どの引数が不正なのかが1テスト1ケースで分かる
- [ ] 例外型だけでなく、必要ならメッセージも確認している
- [ ] バリデーションの責務ごとに分かれている

### 12. E2E / smoke の役割分担

- [ ] e2e テストは配線確認に集中している
- [ ] unit や property と内容が過剰に重複していない
- [ ] smoke テストは最小の import / API 呼び出し確認に留めている

### 13. コメントと説明

- [ ] 数値閾値やドメイン知識が必要な箇所には短い説明がある
- [ ] コメントは「何をしているか」ではなく「なぜこの確認が必要か」を補っている
- [ ] fixture や補助関数の docstring が役割を説明している

## Nice to Have

### 14. ドメイン別の入力パターン整理

- [ ] 線形系列、周期系列、カオス系列、因果あり/なし系列の使い分けが明確である
- [ ] どの系列がどのアルゴリズムのどの性質を見るためのものか説明できる
- [ ] 再利用価値の高い系列は共通 fixture 化されている

### 15. 回帰耐性

- [ ] 過去のバグに対応する回帰テストがある
- [ ] Hypothesis の `@example` などで既知の壊れやすい入力を残している
- [ ] リファクタ時に内部実装変更へ耐えやすい、外部契約ベースのテストになっている

### 16. パフォーマンスと運用

- [ ] 開発時に回しやすい軽量テスト群と、重い検証群が分かれている
- [ ] CI 向けとローカル向けで Hypothesis の強度が整理されている
- [ ] 重い trend test を増やしすぎていない

## EDMKit-Specific Checks

### embedding

- [ ] 出力 shape が正しい
- [ ] 各セルが元系列の正しい lag index に対応している
- [ ] 最小長と不正引数を検証している

### simplex_projection

- [ ] identity query で期待通りの予測になる
- [ ] 定数 target を正しく扱える
- [ ] 予測値が target の値域から大きく外れない
- [ ] ライブラリ順序の入れ替えに対して不変である
- [ ] batch と individual の結果が整合する

### smap

- [ ] `theta=0` の基準ケースを確認している
- [ ] 定数 target や単純系での安定性を確認している
- [ ] 局所性や正則化の影響を、厳密値ではなく相対的傾向で見ている

### ccm

- [ ] helper 関数の unit test が先にある
- [ ] sampler / aggregator / reproducibility を個別に見ている
- [ ] 因果の有無や方向性を trend として確認している
- [ ] 厳密な相関値ではなく、方向や増加傾向を見ている

### generate

- [ ] 出力 shape と有限性を確認している
- [ ] 生成器ごとの定性的制約を見ている
- [ ] 壊れやすい golden trajectory 比較に依存しすぎていない
